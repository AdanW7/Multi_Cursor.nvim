MULTI_CURSOR.NVIM                                        *multi_cursor*

Multicursor plugin for Neovim with two backends:

- legacy (default): embedded vim-visual-multi runtime
- lua: native Lua backend

==============================================================================
SETUP                                                       *multi_cursor-setup*

>lua
require('multi_cursor').setup({
  backend = 'lua',
})
<

To run the embedded upstream runtime:
>lua
require('multi_cursor').setup({
  backend = 'legacy',
})
<

`legacy/vim-visual-multi/doc/`

Quick legacy index:
- Modes: |vm-modes|, |cursor-mode|, |extend-mode|

Legacy quick reference defaults:
- Find under: `<C-n>`
- Next/prev/skip: `n` / `N` / `q`

- Select all: `\\A`
- Regex search: `\\/`
- Slash motion: `g/`
- Toggle mappings: `\\<Space>`
- Toggle mode (cursor/extend): `<Tab>`

- `g:VM_custom_remaps`, `g:VM_custom_noremaps`, `g:VM_custom_motions`
- `g:VM_user_operators`
- `g:VM_live_editing`, `g:VM_use_first_cursor_in_line`

- `g:VM_filesize_limit`, `g:VM_persistent_registers`
- `g:VM_theme`, `g:VM_highlight_matches`

Important legacy Ex commands (see |vm-ex-commands|):
- Always available: `:VMDebug`, `:VMClear`, `:VMRegisters`, `:VMSearch`, `:VMLive`
- Session-scoped: `:VMSort`, `:VMQfix`, `:VMFilterRegions`,
  `:VMFilterLines`, `:VMRegionsToBuffer`, `:VMMassTranspose`
- Lua `:VMSearch` parity notes:
  - `:VMSearch[!] [pattern]` accepts range.
  - Without `!`, selects all matches in given range (default: current line).
  - With `!`, selects next occurrence (pattern or current `@/` register).

Legacy troubleshooting starting points:
- Use |:VMDebug| for mapping conflicts and plugin interference diagnostics.
- Use |:VMClear| to clear leftover highlight matches after failures.
- See |vm-compatibility| for known plugin interaction patterns.

VM LEADER                                                *multi_cursor-leader*

Lua backend supports |g:VM_leader| semantics.

String leader:
>vim
let g:VM_leader = '<leader>m'
<

Lua convenience alias:
>lua
require('multi_cursor').setup({
  multicursor_leader = '<leader>m',
})
<

Per-scope dictionary:
>vim
let g:VM_leader = {'default': '<leader>d', 'visual': '<leader>v', 'buffer': '<leader>b'}
<

Notes:
- `default` scope applies to default/permanent normal mappings.
- `visual` scope applies to visual mappings.
- `buffer` scope applies to buffer-local operational mappings.
- Explicit mapping overrides in `mappings` or `g:VM_maps` are never rewritten.
- `multicursorLeader` is also accepted as a camelCase alias.

LUA MOTION PASSTHROUGH                               *multi_cursor-normal-keys*

If your normal mode motions are remapped (example: `gh`, `gs`, `gl`), include
those keys in `normal_keys` so multicursor normal replay recognizes them.

>lua
require('multi_cursor').setup({
  backend = 'lua',
  enable_normal_key_passthrough = true,
  normal_keys = { 'h', 'j', 'k', 'l', 'gh', 'gs', 'gl' },
})
<

MODES                                                      *multi_cursor-modes*

Lua backend mode transitions follow this model:

- `i` / `a` / `I` / `A` in multicursor normal => multicursor insert mode
- `<Esc>` in multicursor insert => multicursor normal (keeps cursors)
- clear mapping / `:MultiCursorClear` => exits multicursor completely
- toggle mode mapping (default `<Tab>`) switches cursor/extend mode

`cursor` mode:
- each cursor is a point
- operator+motion (`yiw`, `diw`, `ciw`, etc.) first derives selections

`extend` mode:
- each cursor has an anchor + active endpoint
- `y` / `d` / `c` all use the same exact selected range contract
- `d` / `c` yank exact selected text first, then delete/change

Search-seed workflow (Lua backend):

- `seed_word_search` (default `\<leader>*`, with `multicursor_leader='<leader>m'` => `<leader>m*`)
  seeds pattern from word under cursor and starts multicursor session
- then `find_next` / `find_prev` (defaults `n` / `N`) add cursors at next/prev matches
- `skip` (default `q`) skips the current selection and advances to the next match

REGISTERS                                              *multi_cursor-registers*

Lua backend register behavior:

- Uses current register (`"` by default, `vim.v.register`)
- Supports named registers (`"a` ... `"z`)
- Supports append registers (`"A` appends into `"a`)
- Supports black-hole register (`"_"`)
- `d`/`c` with unnamed register rotate delete registers (`"1` ... `"9`)
- Mirrors multicursor yanks into Vim registers via `setreg()`

Paste source priority:
- multicursor internal register
- fallback to native Vim register contents

Note:
- Numbered register behavior follows common Vim expectations for unnamed
  delete/change operations in Lua backend.

==============================================================================
COUNTS                                                    *multi_cursor-counts*

Lua backend count behavior (current status):

Supported:
- Operator prompts with motions in active multicursor mode (`2dw`, `3yw`, etc.)
- Paste mappings in active multicursor mode (`2p`, `2P`)
- Numeric commands already designed for counts (numbers/regex/goto-region flows)

Intentional limits (current):
- Not every command-specific motion flow has bespoke count parity with legacy VM.
- If a command path has no explicit count contract, behavior follows its mapped
  action once.

Recommendation:
- Prefer explicit count-enabled mappings/commands above.
- For advanced flows, verify behavior with `:checkhealth multi_cursor` and the
  parity tests listed in `tests/test_backend_parity_subset.py`.

==============================================================================
CASE ADJUSTMENT                                    *multi_cursor-case-adjust*

Case tools in Lua backend:

- `case_conversion` mapping / `M.case_convert(mode)` applies to selected text.
- In cursor mode, case conversion first derives `iw` selections at each cursor.
- In extend mode, conversion applies to current selected regions.
- After conversion, mode returns to `cursor` (legacy-compatible behavior).

Register/selection interaction:

- Case conversion does not intentionally write to delete/yank registers.
- Selection probing used to derive cursor-mode text objects restores unnamed
  register content after probing.

Case-setting cycle:

- `case_setting` mapping / `M.case_setting_cycle()` rotates:
  `smart` -> `sensitive` -> `ignore` -> `smart`.
- Startup `case_setting` option still controls search-case defaults when
  multicursor session starts.
